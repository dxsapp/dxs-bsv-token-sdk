### 2.2 Gate G2 — Whitelist commitment soundness

**Verdict: PASS.**

S2 defended 5/5 load-bearing claims and addressed 4/4 spec-enumerated attack surfaces. Additionally enumerated 11 new attack surfaces, all dismissed with rigorous argument. Zero unmitigated findings.

Key insight validated: `h_X = SHA256(PREFIX_X ‖ SUFFIX_X)` excludes WHITELIST block, making the substitution function a **constant function** (not a true fixpoint equation). Constant functions have exactly one fixed point, eliminating circularity. Soundness reduces to SHA256 preimage/collision resistance + byte-exact OP_EQUALVERIFY.

No NEW ATTACK SURFACE DISCOVERED flag raised.

### 2.3 Gate G3 — Anchor/follower algorithm security

**Verdict: PASS.**

S3 designed concrete algorithm with opcode sketches. Defended 7/7 attack scenarios:

- A1 input reorder — caught by hashPrevouts binding of `all_outpoints` push
- A2 anchor-spoofing via forged `all_outpoints` — caught by HASH256 check
- A3 follower-delegation escape (with 3 sub-attacks) — caught by reconstruction byte-match forcing input[0] to be a real Normal UTXO → Bitcoin consensus guarantees its script executes
- A4 mixed-token merge — caught by tail field equality check
- A5 K-lie attack (3 sub-variants) — caught by followerCount cross-check + reconstruction
- A6 multiple-anchors — no two inputs at same position, handled by position computation
- A7 funding-at-position-0 — caught by reconstruction (funding P2PKH has no valid Normal body)

5/5 edge cases covered (K=2, K=4, funding position, cross-series, K=0/K=1).

Key design insight validated: `selfPosition` is **never trusted from unlocking** — it's computed by scanning hashPrevouts-bound `all_outpoints` at preimage-offset 68 for self-outpoint.

## 3. What residuals remain

### 3.1 SPEC AMENDMENT REQUESTs (total 11 across three slices)

**From S1 (3):**

1. §9.4 issuer attestation mechanism — current spec implies CHECKSIG-over-arbitrary-hash which doesn't exist natively in BSV Script. Redesign proposal: encode attestation as required null-data output `OP_FALSE OP_RETURN <tokenId> <thisOutpoint> <timestamp>` covered by issuer's sighash-preimage signature. Estimated savings: ~200b in Normal path 3.
2. §9.2 / §11 unlocking format — current tuple form `[satoshis, owner, action_data, body_marker]` insufficient for whitelist + body-hash verification. Spec should be explicit: unlocking pushes FULL candidate locking script bytes, SDK builder derives from components.
3. §5 / §7 — lock normative rule: WHITELIST always at offset `[|PREFIX|, |PREFIX|+96)` relative to body start.

**From S2 (4, all non-breaking clarifications):** 4. §5.3 — explicit that 96-byte zero placeholder is never a hash input (for pseudo-ASM clarity). 5. §7.2 — `h_candidate` hashes raw byte slices, not canonicalised opcodes (pushdata-malleability note). 6. §7.2 / §7.4 — series templates accept only `{0x01ff, 0xfeff, 0x0fff}` body markers for output verification; `0xc0ff` (Contract) is rejected for series-emitted outputs; Contract's issue-path accepts only `0x01ff` (Normal). 7. Optional: Contract issue-path may add on-chain check `this.tail.seriesId == SHA256(this.body.WHITELIST)` (~40b) to catch deployer-mis-stated seriesId.

**From S3 (4):** 8. §9.3 — explicit input layout invariant: BNTP inputs contiguous at positions 0..K-1, funding strictly at K. Resolves attack A7 and edge case §5.3. 9. §9.3 / §11 unlocking catalog — clarify that `all_outpoints` (K×36b, BNTP only) and `funding_outpoint` (36b) are separate pushes, hash check over concatenation in that order. 10. §9.3 anchor path — require redundant cross-check `followerCount == (|all_outpoints|/36) - 1` (cheap, resolves A5). 11. Future `BNTP_INVARIANTS.md` — lock BSV BIP143 preimage offsets used by algorithm (outpoint=68, hashPrevouts=4).

### 3.2 OPEN QUESTIONs (total 5 from S1)

1. Body-marker encoding: push+drop (5b) vs inline non-executable bytes.
2. WHITELIST encoding: push+drop (99b) vs three 32-byte pushes (102b) vs pre-OP_RETURN inline.
3. Can §5.2 share reconstruction helper between anchor and follower via OP_IF fall-through? Potential savings ~200b.
4. §5.3 / path 3 issuer attestation mechanism — **resolvable via SPEC AMENDMENT REQUEST #1 above.**
5. Royalty minimum encoding: embedded constant (9b) vs spec parameter.

### 3.3 Budget re-baseline implications

If we accept the S1 pivot proposal (revise Normal body budget to ~3000b, split prepare-swap into dedicated `NormalSwapOnRamp` template), the updated whitelist grows to **4 templates** (NormalBase + NormalSwapOnRamp + Frozen + SwapReady):

- WHITELIST block: 96b → 128b (+32b per template)
- seriesId computation: still SHA256(WHITELIST) — same primitive
- Output verification: check h_candidate ∈ 4 hashes instead of 3

Alternative: drop prepare-swap entirely from a template and require external "swap contract" (not part of BNTP v1). This keeps whitelist at 3 but requires users to use a separate protocol for DEX entry.
